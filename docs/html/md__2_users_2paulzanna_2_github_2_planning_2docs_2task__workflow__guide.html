Task Workflow Guide — Using the Life‑Plan Repo

This guide explains how to use GitHub Issues, Projects, and Pull Requests to manage and track all work in your Life‑Plan repository.

1. Overview

The repo uses a docs‑as‑code setup with GitHub Projects (Beta). Each piece of work lives as an Issue (Task, Epic, Risk, or Decision).
PRs reference Issues using Fixes #<issue_number> so progress automatically syncs with your Project board.

2. Initial Setup Checklist

  1. Create a GitHub Project (Beta) for this repo.
  2. Add repo secrets under Settings → Secrets → Actions:
    • PROJECT_PAT — fine‑grained token with project + issues + repo write.
    • GH_PROJECT_URL — URL of your Project (e.g. https://github.com/users/PaulZanna/projects/1).
  3. Ensure the following custom fields exist in the Project:
    • Status → Backlog, In Progress, Blocked, In Review, Done
    • Area → Eunice, Ludo, Brain, Scout, Writing, Ops
    • Priority → P0, P1, P2, P3
    • Size → XS, S, M, L, XL
    • Gate → E1, S0, etc. (free text)
    • Owner (text), KPI (text), Due (date)
  4. Pin Project views:
    • Sprint Board — Group by Status, filter -status:Done
    • Roadmap — Timeline grouped by Area
    • Risk Register — Table filter label:risk/*

3. Creating Work

3.1 Tasks

  • Use the Task Issue template (from repo → New issue → Task).
  • Keep tasks 1–3 days in size.

  • Example:

    task: Eunice – container skeletons and health checks
    Area: Eunice
    Priority: P1
    Due: 2025‑11‑07
    Steps:
    - [ ] scaffold service template
    - [ ] add make‑based generator
    - [ ] implement healthcheck endpoints

3.2 Epics

  • Use the Epic template to group multiple tasks.

  • Example:

    epic: Eunice v1 (Gate E1 – reliability + 5 pilot users)
    Definition of done: reliability suite passes, 5 pilots onboarded.
    Tasks:
    - [ ] #12 Eunice – container skeletons
    - [ ] #14 Eunice – MCP wiring

3.3 Risks

  • Use the Risk template to document threats and mitigations.
  • Label automatically: risk/needs‑triage.

  • Example:

    risk: UPS backup rotation untested
    Severity: M
    Mitigation: add quarterly restore‑test drill.

3.4 Decisions

  • Capture important design or strategic calls.

  • Example:

    decide: Ludo actuator material
    Context: weight vs durability trade‑off
    Decision: use 7075‑T6 aluminium
    Follow‑up: update BOM + docs.

4. Linking Code to Issues (GitHub Desktop)

  • Create a branch:
    • Open GitHub Desktop with this repo selected.
    • Click Current Branch → New Branch…, name it feat/eunice/health-checks, base main.
    • Click Create Branch to check out the new branch.
  • Make changes and commit:
    • Edit files in your editor; changes appear in GitHub Desktop.
    • Write a concise Summary (e.g., feat(eunice): add health checks).
    • Optionally add details in Description.
    • Click Commit to feat/eunice/health-checks.
  • Publish/push the branch:
    • First time: click Publish branch.
    • After subsequent commits: click Push origin.
  • Open a Pull Request:
    • Click Create Pull Request (or Open on GitHub → Compare & pull request).
    • In the PR title or body, include Fixes #12 to link and auto‑close the Issue.
    • Add reviewers/labels; Project automation will attach it to the board.
  • Merge and close:
    • After review and checks pass, merge the PR on GitHub.
    • Issue #12 auto‑closes and moves to Done.

5. Weekly Flow

Day Action
Sunday Review board, pick 3–5 Tasks, set Status=In Progress.
Mon–Thu AM Deep work on selected Tasks (Eunice/Ludo).
Thursday PM Move completed Tasks to In Review.
Friday Review demos, close or re‑scope blocked work.
Saturday Lab block for Ludo/Bio/Scout work.

6. Monthly & Quarterly Rhythm

  • Month‑end: roll incomplete work forward; clean up stale Issues.
  • Quarterly: automation opens a Quarterly Review Issue.
    • Review KPIs (health, finance, work, resilience, joy).
    • Evaluate §13 indicators in the life plan (Ferguson limit, BRICS/mBridge, etc.).
    • Apply Stop/Continue/Pivot gates for each Epic.

8. Typical Seeds

  1. Epic: Eunice v1 (Gate E1 – reliability + 5 pilot users)
    • Tasks: containers, MCP wiring, DB read/write, eval harness, pilot onboarding.
  2. Task: Ludo – actuator v3 bearing selection (2025Q4)
    • Steps: load calc, shortlist bearings, supplier quotes, sample order.
  3. Risk: Backup restore test missed → Mitigation: add to quarterly checklist.

9. Automation Checks

  • Add‑to‑Project → confirms Issues/PRs land on board.
  • Require Linked Issue → PR fails if missing Fixes #number.
  • Quarterly Review → new issue auto‑opens each quarter with agenda.

10. Quick Troubleshooting

Problem Likely cause Fix
Issue not on Project Wrong GH_PROJECT_URL or token scope Verify secrets & Action logs
PR didn’t close Issue Missing or mistyped Fixes #number Edit PR body or merge manually
Labels missing Didn’t run setup script bash scripts/gh-setup.sh

11. Best Practices

  • Limit yourself to ≤2 active Epics at once.
  • Prefer closing small Tasks fast over big all‑at‑once efforts.
  • Write Issues in action language (“build”, “refactor”, “test”).
  • Keep Project views clean: archive Done cards monthly.

TL;DR

  1. File a Task or Epic → automation adds it to board.
  2. Link PRs with Fixes #id.
  3. Work, review, close.
  4. Quarterly issue opens → measure, decide, move on.
