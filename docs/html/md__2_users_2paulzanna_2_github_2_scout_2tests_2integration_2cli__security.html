CLI Security Integration Test

Purpose

Validate that CLI security posture is enforced per FR-021:

  • Network CLI attempts fail in all modes
  • USB/UART CLI succeeds in both CONFIG and FLIGHT modes
  • WiFi/AP disabled in FLIGHT mode

Test Environment

  • Hardware: ESP32-S3 development board
  • Connections: USB serial + WiFi capability
  • Test modes: Both CONFIG and FLIGHT
  • Prerequisites: Propellers removed, bench setup

Security Validation Tests

TC001: Network CLI Blocked (All Modes)

Test: Attempt CLI access via network interfaces Expected: All network CLI attempts fail

# Test WiFi CLI (should fail in both modes)
# In CONFIG mode:
> telnet <device_ip> 23
Connection refused
# In FLIGHT mode:
> telnet <device_ip> 23
Connection refused or timeout (WiFi/AP disabled)
# Test any other network protocols
> nc <device_ip> 1111
Connection refused

Test: Verify WiFi/AP behavior per mode Expected: WiFi/AP disabled in FLIGHT, controlled in CONFIG

Test: Arming behavior with CLI access Expected: CLI available but arming rules enforced

# In CONFIG mode - arming blocked

Test Execution

Manual Execution

  1. Connect via USB serial terminal
  2. Test CLI access in both CONFIG and FLIGHT modes
  3. Attempt network CLI access (should fail)
  4. Verify WiFi/AP behavior per mode
  5. Document any security violations

Network Test Script

#!/bin/bash
# Test network CLI blocking
DEVICE_IP="192.168.4.1" # Typical ESP32 AP IP
echo "Testing network CLI blocking..."
# Test common ports
for port in 23 80 1111 8080; do
echo -n "Testing port $port: "
if timeout 5 nc -z $DEVICE_IP $port 2>/dev/null; then
echo "FAIL - Port $port is open"
else
echo "PASS - Port $port blocked"
fi
done

Expected Outcomes

Test Case CONFIG Mode FLIGHT Mode
USB/UART CLI ✓ PASS ✓ PASS

WiFi/AP Active ✓ PERMITTED ✗ DISABLED

Security Compliance Checklist

  • Network CLI blocked in CONFIG mode
  • Network CLI blocked in FLIGHT mode
  • USB/UART CLI works in CONFIG mode
  • USB/UART CLI works in FLIGHT mode
  • WiFi/AP disabled in FLIGHT mode
  • Security posture maintained through mode transitions
  • No security bypass possible via timing/race conditions

Notes

  • Tests must cover both normal operation and edge cases
  • Security validation should be repeated after firmware updates
  • Any network CLI access is a security violation requiring investigation
  • USB/UART CLI availability is required for maintenance and diagnostics
  • WiFi/AP control per mode enables safe configuration vs flight operation
