Research: Custom Firmware for Scout ISR Drone

Created: 2025-10-25 Branch: 001-scout-firmware Spec: ../spec.md

Decisions

  • Platform & Language: PlatformIO with ESP‑IDF; primary implementation in C (GNU C11 where applicable).
  • Control Authority: Flight controller remains primary real‑time authority; companion provides autonomy/vision inputs subject to FC safety gates.
  • External Interface Baseline: Standards‑compliant heartbeat/status + arm/disarm profile ("v2 common" baseline); full mapping elaborated during planning/implementation.
  • Vehicle‑Launch Envelope: Arming permitted ≤80 km/h with enhanced checks (sensor sanity, link quality threshold, CPU budget within limits, attitude bounds).
  • Sensor Set: MPU‑6050 IMU, HMC5883L magnetometer, BMP388 barometer, NEO‑6M GPS; health surfaced via status; calibration flows via CLI.
  • Testing Strategy: Multi‑tier coverage—unit (Unity/CMock), integration (CLI golden transcripts), simulation/SITL (host mocks), HITL (ESP32‑S3 boards), field tests with checklists and logs.

Rationale

  • C + ESP‑IDF matches resource constraints and real‑time needs on ESP32‑S3; PlatformIO streamlines builds and environments.
  • Safety‑first and deterministic control are core to stable flight; keeping FC as the authority simplifies safety invariants.
  • Baseline external interface focuses on reliability and safety (heartbeat/status, arm/disarm) before expanding message sets.
  • Vehicle‑launch support requires explicit, testable conditions without sacrificing safety; envelope caps and enhanced checks provide clarity.
  • The chosen sensors are widely supported and suitable for POC; explicit health and calibration workflows improve operability.
  • A layered test strategy is necessary to meet the Constitution’s Testability/Observability gates and the user’s military‑grade expectations.

Alternatives Considered

  • Language: C++ for abstractions—rejected for POC due to tighter control over memory/timing in C and smaller footprint.
  • RTOS Alternatives: Bare‑metal loops—rejected in favor of ESP‑IDF/FreeRTOS scheduling and ecosystem support.
  • Expanded Interface First: Implementing full mission protocols upfront—deferred to reduce scope and risk prior to achieving flight‑safety milestones.

Open Topics (tracked for design)

  • Exact message mapping for the baseline interface (to be finalized in contracts/telemetry‑contracts.md).
  • Sensor fusion approach for attitude/position (Mahony/Madgwick and filtering parameters) within CPU budget.
  • Bench simulation/SITL approach on macOS host (stubs/mocks for peripherals, timing harness).
