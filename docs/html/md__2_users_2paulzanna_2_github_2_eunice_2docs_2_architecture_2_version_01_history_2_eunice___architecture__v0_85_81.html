Eunice v0.5.1 — DB‑Driven Research Manager (RM) Spec

Status: Implementation spec • Date: 2025‑08‑10 • Scope: RM as standalone container, no public workflow API. RM reacts to DB state (polls + leases), writes back results. First workflow: Create Research Plan from Topic.

1) Goals & Constraints

  • RM runs as independent container (a.k.a. orchestrator).
  • No RM HTTP workflow endpoint. All orchestration is DB‑triggered.
  • Triggers via inbox table (recommended) and/or status flags.
  • Exactly‑once-ish execution using row leases + idempotency keys.
  • First workflow delivers: topic → research_plan + seed research_tasks in a single transactional write.

References: See platform architecture docs (v0.5.x) for MCP client, db_query/db_mutate, and service standards.

2) Database Changes

2.1 Inbox table (primary trigger)

CREATE TABLE IF NOT EXISTS rm_inbox (
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
type TEXT NOT NULL, -- e.g. 'create_research_plan'
entity_type TEXT NOT NULL, -- e.g. 'research_topic'
entity_id UUID NOT NULL, -- FK to research_topics.id
payload JSONB NOT NULL DEFAULT '{}'::jsonb,
idempotency_key TEXT UNIQUE, -- optional; client dedupe
status TEXT NOT NULL DEFAULT 'queued', -- queued|running|done|failed
attempt_count INTEGER NOT NULL DEFAULT 0,
not_before TIMESTAMPTZ NOT NULL DEFAULT now(),
leased_by TEXT,
leased_at TIMESTAMPTZ,
lease_expires_at TIMESTAMPTZ,
result_ref JSONB,
error_info JSONB,
created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_rm_inbox_status_notbefore
ON rm_inbox (status, not_before);
CREATE INDEX IF NOT EXISTS idx_rm_inbox_entity
ON rm_inbox (entity_type, entity_id);

Update trigger timestamp: implement updated_at via DB default or app‑side.

2.2 Optional implicit flag

ALTER TABLE research_topics
ADD COLUMN IF NOT EXISTS plan_desired BOOLEAN NOT NULL DEFAULT FALSE;

2.3 Topic uniqueness guard (avoid duplicate draft plans)

CREATE UNIQUE INDEX IF NOT EXISTS uq_topic_single_draft_plan
ON research_plans (topic_id)
WHERE status IN ('draft','proposed','pending_review');

3) Enqueue Patterns (Gateway responsibility)

3.1 Inbox mode (preferred)

INSERT INTO rm_inbox (type, entity_type, entity_id, payload, idempotency_key)
VALUES (
'create_research_plan',
'research_topic',
:topic_id,
coalesce(:payload, '{}'::jsonb),
:idempotency_key
)
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING id;

Typical payload (options / overrides):

{
"scope": "Comprehensive",
"llm": {"model":"grok-3-mini","temperature":0.4,"max_tokens":3000},
"task_defaults": {"assignee_id": null, "priority": "normal"}
}

3.2 Flag mode (simple)

UPDATE research_topics SET plan_desired = TRUE WHERE id = :topic_id;

4) RM Poller & Lease Protocol

4.1 Leasing query

WITH cte AS (
SELECT id
FROM rm_inbox
WHERE status='queued'
AND not_before <= now()
AND (lease_expires_at IS NULL OR lease_expires_at < now())
ORDER BY created_at
FOR UPDATE SKIP LOCKED
LIMIT :batch_size
)
UPDATE rm_inbox i
SET status='running',
leased_by=:instance_id,
leased_at=now(),
lease_expires_at=now() + (:lease_seconds || ' seconds')::interval,
attempt_count = i.attempt_count + 1,
updated_at=now()
FROM cte
WHERE i.id = cte.id
RETURNING i.*;

4.2 Heartbeat (extend lease while running)

UPDATE rm_inbox
SET lease_expires_at = now() + (:lease_seconds || ' seconds')::interval,
updated_at = now()
WHERE id = :job_id
AND leased_by = :instance_id;

4.3 Backoff on transient failure

  • Next attempt: not_before = now() + INTERVAL '1 second' * LEAST(60, POWER(2, attempt_count)).
  • Cap attempts (e.g., MAX_ATTEMPTS = 7) then park as failed.

4.4 Finalization

-- success
UPDATE rm_inbox
SET status='done', result_ref=:result_json, updated_at=now()
WHERE id=:job_id AND leased_by=:instance_id;
-- failure
UPDATE rm_inbox
SET status='failed', error_info=:error_json, updated_at=now()
WHERE id=:job_id AND leased_by=:instance_id;

5) First Workflow — Create Research Plan from Topic

5.1 Preconditions

  • research_topics.id = entity_id exists and is accessible.
  • No existing draft/proposed/pending plan for this topic (unique index).
  • If flag mode: plan_desired = TRUE.

5.2 Prompt contract (JSON‑only)

Prompt file: research-manager/prompts/create_research_plan.json
LLM output schema (strict):

{
"type": "object",
"required": ["name", "objectives", "methodology", "seed_tasks"],
"properties": {
"name": {"type": "string", "minLength": 3},
"summary": {"type": "string"},
"objectives": {"type": "array", "items": {"type": "string"}, "minItems": 1},
"methodology": {
"type": "object",
"required": ["design","data_sources","analysis_methods","risks","timeline"],
"properties": {
"design": {"type":"string"},
"data_sources": {"type":"array","items":{"type":"string"}},
"analysis_methods": {"type":"array","items":{"type":"string"}},
"risks": {"type":"array","items":{"type":"string"}},
"timeline": {
"type":"object",
"properties": {
"milestones": {"type":"array","items":{"type":"string"}}
},
"required": ["milestones"]
}
}
},
"seed_tasks": {
"type": "array",
"minItems": 2,
"items": {
"type":"object",
"required":["name","type"],
"properties": {
"name":{"type":"string"},
"type":{"type":"string", "enum":["literature_search","screening","analysis","writing","other"]},
"metadata":{"type":"object"}
}
}
},
"references": {"type":"array","items":{"type":"string"}}
}
}

Validation: JSON Schema (fastjsonschema/ajv equivalent). If invalid, run one repair attempt; otherwise mark job failed.

5.3 DB mutation (single transaction)

Use unified MCP tool db_mutate with idempotency key topic-{topic_id}-plan-v1.

Mutation payload (conceptual):

{
"operation": {
"type": "create",
"resource": "research_plans",
"data": {
"topic_id": "<topic_uuid>",
"name": "{{ llm.name }}",
"summary": "{{ llm.summary | default('') }}",
"methodology": {{ llm.methodology }},
"status": "draft",
"approval_status": "proposed"
},
"capture_as": "PLAN_ID"
},
"operations": [
{% for t in llm.seed_tasks %}
{
"type": "create",
"resource": "research_tasks",
"data": {
"plan_id": "$PLAN_ID",
"name": "{{ t.name }}",
"type": "{{ t.type }}",
"status": "pending",
"execution_details": {
"source": "auto-seed",
"metadata": {{ t.metadata | default({}) }}
}
}
}{% if not loop.last %},{% endif %}
{% endfor %}
],
"options": {
"tx": true,
"idempotency_key": "topic-<topic_uuid>-plan-v1"
}
}

Success result_ref example:

{"plan_id":"<uuid>","task_ids":["<uuid1>","<uuid2>", "..."]}

5.4 Idempotency rules

  • Client‑side dedupe: unique idempotency_key on rm_inbox.
  • Writer dedupe: db_mutate.options.idempotency_key ensures stable replay.
  • Schema guard: unique index prevents multiple drafts for same topic.

5.5 Flag cleanup (if used)

UPDATE research_topics SET plan_desired = FALSE WHERE id = :topic_id;

8) Minimal API Helper (Gateway)

Note: This endpoint only writes to DB to enqueue work; it does not call the RM.

POST /topics/{topic_id}/ai-plans:enqueue (legacy: /v2/...)
Body: {
"scope":"Comprehensive",
"idempotency_key":"topic-<topic_id>-ai-plan-v1",
"llm":{"model":"grok-3-mini","temperature":0.4,"max_tokens":3000}
}
→ Gateway executes INSERT into rm_inbox (or sets plan_desired flag). Returns 202 with { "job_id": "<uuid>" }.

9) Test Plan (v0.5.1)

  • Inbox enqueue is idempotent (conflicting idempotency_key → single row).
  • Lease contention: two RM instances → exactly one executes (SKIP LOCKED + lease).
  • Heartbeat: long‑running job extends lease; no double run.
  • Transient failure: backoff & retry; success on second attempt marks done once.
  • Duplicate enqueue: multiple rows → unique draft index + writer idempotency ⇒ one plan.
  • Atomicity: simulated mid‑batch failure rolls back entire transaction.
  • Flag mode: setting plan_desired=TRUE leads to plan creation, then flag cleared.
  • Observability: metrics increment correctly; logs contain traceable keys.

10) Rollout Steps

  1. Apply DB migration(s) for rm_inbox, plan_desired, and unique draft index.
  2. Ship RM container with poller + create_plan workflow.
  3. Add Gateway helper endpoint to enqueue jobs.
  4. Configure env defaults; deploy with 1 replica, then scale horizontally.
  5. Run test plan in staging; monitor metrics; promote to prod.

11) Appendix — Example RM Worker Pseudocode

job = lease_one()
if not job:
return
with trace_context(job.id):
try:
topic = db_query_topic(job.entity_id)
plan_json = llm_generate_and_validate(topic, job.payload) # strict JSON
mutation_payload = build_mutation(topic.id, plan_json)
result = db_mutate(mutation_payload, idempotency_key=f"topic-{topic.id}-plan-v1", tx=True)
record_success(job.id, result) # status=done, result_ref=...
clear_flag_if_any(topic.id)
except IdempotencyCollision as e:
record_success(job.id, e.existing_result)
except TransientError as e:
schedule_retry(job.id, e) # backoff + not_before
except Exception as e:
record_failure(job.id, e) # status=failed + error_info
