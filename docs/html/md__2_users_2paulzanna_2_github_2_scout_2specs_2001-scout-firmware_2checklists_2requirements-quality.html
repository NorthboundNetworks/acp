Requirement Completeness

  • CHK001 Are safety gate requirements and preflight checklist steps fully enumerated (inputs, pass/fail rules, operator feedback)? [Completeness, Spec §FR-001, §FR-010]
  • CHK002 Are all primary operation modes (hover, loiter, forward) defined with constraints and transitions? [Completeness, Spec §FR-008]
  • CHK003 Is the baseline external interface message set (heartbeat/status, arm/disarm) concretely mapped (IDs, fields, rates)? [Completeness, Spec §FR-012, Contracts]
  • CHK004 Are calibration workflows for each applicable sensor (MAG/IMU) described with steps, prompts, and success criteria? [Completeness, Spec §FR-018]
  • CHK005 Is a degrade policy specified per sensor failure case (what to restrict, when to disarm, how to record)? [Completeness, Spec §FR-020]

  • CHK007 Are configuration import/export boundaries and size/format assumptions documented? [Completeness, Spec §FR-004]

Requirement Clarity

  • CHK008 Is "telemetry ≥1 Hz" defined with required fields and acceptable jitter/drop tolerance? [Clarity, Spec §FR-006, §SC-003]
  • CHK009 Are loop rate targets (±5%) tied to named loops and measurement methodology? [Clarity, Spec §FR-005, §SC-004]
  • CHK010 Is link‑loss detection threshold and timer source specified for the ≤1s disarm criterion? [Clarity, Spec §FR-002, §SC-006]
  • CHK011 Are vehicle‑launch enhanced checks (≤80 km/h, ±15°) defined with data sources, windowing, and tolerances? [Clarity, Spec §FR-013]
  • CHK012 Is "CPU ≤70%" defined with sampling window, core scope, and what counts as flight CPU? [Clarity, Spec §FR-005, §SC-004]
  • CHK013 Is the security posture stated unambiguously for all modes (CONFIG vs FLIGHT) and interfaces (USB/UART vs WiFi/AP)? [Clarity, Spec §FR-021, Clarifications]
