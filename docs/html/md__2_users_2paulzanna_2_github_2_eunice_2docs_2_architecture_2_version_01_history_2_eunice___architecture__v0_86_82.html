Eunice v0.6.2 Architecture — Agent Runtime & Scheduling

Date: 12-08-2025
Owner: Paul Zanna
Status: Implementation-Ready

1) Executive Summary

v0.6.2 introduces a resumable, idempotent job runtime with a DAG scheduler, queue, and a Git‑backed artefact store. Runs are captured as manifests; failed stages can resume; caching and dependency awareness reduce cost and latency.

2) Scope

Goals

  • DAG execution engine with retries/backoff and per‑stage caching.
  • Schedules (cron‑like) and manual triggers.
  • Git‑backed artefact store and immutable run IDs.
  • Run manifests with hashes of inputs/prompts/models/tools.

Non‑Goals (now)

  • Distributed cluster across many machines (single host, multi‑process acceptable).

3) High‑Level Architecture

flowchart LR
A[Job API/CLI] --> B[Planner -> DAG]
B --> C[Scheduler]
C --> D[Worker Queue]
D --> E[Stage Executors]
E --> F[Artifact Store (Git)]
E --> G[Cache]
E --> H[Run Manifest Writer]
I[Retry/Backoff Policies] --> C

Key properties

  • Idempotent stages; deterministic cache keys.
  • Resume from last successful stage.

4) Components

4.1 Planner

  • Translates high‑level tasks into DAGs with explicit inputs/outputs.

4.2 Scheduler & Queue

  • Time‑based triggers; dependency resolution; concurrency control.

4.3 Executors

  • Stage runners with sandboxed environment, seed control, and cache integration.

4.4 Artefact Store

  • Git repository for artefacts with LFS optional; tags per run; provenance files.

6) Interfaces

6.1 CLI

  • eunice job create --plan <file>
  • eunice job run --id <job_id>
  • eunice job schedule --id <job_id> --cron "0 9 * * 1"
  • eunice job status --id <job_id>
  • eunice job resume --id <job_id>

6.2 Internal FastAPI

POST /jobs -> {plan} -> {job_id}
POST /jobs/{id}/run
POST /jobs/{id}/schedule -> {cron}
GET /jobs/{id} -> {status, tasks}
POST /jobs/{id}/resume

7) Observability

  • Artefacts: run manifests, logs, DAG graphs, cached blobs.
  • Metrics: task success rate, retries, cache hit ratio, runtime per stage.

8) Testing & CI

  • Idempotency tests (re‑run equals cached outputs).
  • Failure injection (stage N fails, resume continues).
  • Concurrency & ordering tests.

9) Acceptance (DoD)

  • Jobs resume after failure; cache prevents recomputation.
  • Artefacts committed to Git; manifests complete.
