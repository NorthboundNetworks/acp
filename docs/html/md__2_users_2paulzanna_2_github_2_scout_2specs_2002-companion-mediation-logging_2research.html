Research & Decisions

Date: 2025-10-26
Feature: specs/002-companion-mediation-logging/spec.md

1) Storage format and medium for mission logs

  • Decision: FATFS partition with NDJSON line-oriented files; keep-last-1 rotation.
  • Rationale: NDJSON streams efficiently, supports partial reads and simple export; FATFS is mature on ESP-IDF and better suited than raw NVS blobs for append + export.
  • Alternatives considered:
    • NVS blobs: simple key/value but poor for large appendable logs and export.
    • SPIFFS/littlefs: viable, but FATFS benefits from existing ESP-IDF tooling and binary compatibility with host parsing.

2) Export integrity/authenticity

  • Decision: HMAC-SHA256 signature over entire NDJSON content; include {algo, keyLabel, signature} in export metadata.
  • Rationale: Ensures tamper detection and provenance with low compute cost on MCU.
  • Alternatives considered: SHA256 checksum only (no authenticity), CRC32 (weak), full public-key signing (heavier key mgmt on MCU).

3) HMAC key management

  • Decision: Store key in NVS under operator-defined key label; set via CLI hmac set <label> <hexKey>; show label (not key) via hmac show.
  • Rationale: Keeps secret out of transcripts; manageable in the field; supports rotation by new label and re-export.
  • Alternatives considered: Hardcoded key (unacceptable), companion-provided per-export key (operationally complex), secure element (future option).

4) Telemetry audit mapping

  • Decision: Use SCP audit telemetry for audit summaries (rejections, arming/mode changes), with 2 msgs/sec cap and 500 ms coalescing.
  • Rationale: Native SCP path with rate limiting preserves bandwidth and simplifies integration.
  • Alternatives considered: Custom SCP message types (heavier), no telemetry summaries (reduced situational awareness).

5) Gimbal control interface

  • Decision: Use SCP gimbal command/control mapping; expose CLI with modes: fixed/follow/sweep.
  • Rationale: Consistent with SCP-only runtime; clear operator feedback over the same protocol.
  • Alternatives considered: Custom vendor commands; direct PWM-only (less interoperable).

6) Companion input trust model

  • Decision: Single whitelisted companion identity enforced at input gateway; all others rejected and logged.
  • Rationale: Minimizes attack surface and ambiguity; aligns with safety-first policy.
  • Alternatives considered: Multi-source arbitration (complex), open acceptance with safety-only gating (weaker).

7) Sensor summary cadence

  • Decision: Every 2 seconds by default.
  • Rationale: Balance between fidelity and storage/CPU; sufficient for post-flight narrative.
  • Alternatives considered: 1s (heavier), 5-10s (too sparse for ISR review).

8) NDJSON event schema

  • Decision: Flat top-level with ts, type, severity?, and attrs (object); include schema_version per record.
  • Rationale: Keeps entries flexible while maintaining basic invariants.
  • Alternatives considered: Deeply nested schema (harder to stream), CSV (poor fit for nested attributes).

9) File I/O and real-time constraints

  • Decision: Use a background writer task with bounded queue; drop non-critical summaries if queue is full; never block control loops.
  • Rationale: Preserves real-time guarantees; graceful degradation.
  • Alternatives considered: Synchronous writes (risky), unbounded buffers (memory risk).
