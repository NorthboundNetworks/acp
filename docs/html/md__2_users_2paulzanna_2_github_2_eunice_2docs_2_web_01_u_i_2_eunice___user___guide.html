Eunice Web UI User Guide

Current as of 0.6.4 (read‑only & workflow scaffolds)

1. Overview

The Eunice Web UI provides a secure, minimal interface for:

  • Managing Projects, Topics, Plans & Tasks
  • Generating & approving AI research plans
  • Monitoring system health & task lifecycle
  • Viewing protocol playbooks, materials & equipment (early preview)
  • Managing user profile & two‑factor authentication (2FA)

The UI is intentionally lean; advanced operations remain CLI‑first.

2. Access & Authentication

  1. Navigate to http://localhost:3000 (default dev port)
  2. Register or log in (JWT + refresh managed by API Gateway)
  3. (Optional) Enable 2FA under Profile → Two‑Factor Setup

Session cookies are http‑only; manual token handling is not required.

3. Navigation Structure

Route groups (protected unless noted):

  • /auth — Login / registration (public)
  • /app (implicit via protected routing)

4. Projects & Topics

Create a project:

  1. Go to Projects → New Project.
  2. Supply name & optional description; submit.
  3. Accordion entries list projects; expand for metadata.

Edit / delete via icon buttons on each project row. Navigate to a project to view (or soon manage) topics. Topic CRUD follows the same pattern (creation UI appears in project detail page in upcoming iteration; for now use CLI if absent).

5. Plans

Two creation modes:

  • Manual (POST /topics/{id}/plans)
  • AI Generated (Generate Plan button calling /topics/{id}/ai-plans)

After review, click Approve (POST /plans/{id}/approve). Approved plans expose task lifecycle buttons and downstream artefact views (materials, equipment, protocols).

6. Task Lifecycle

States: pending → running → completed | failed. Buttons (Start / Complete / Fail) invoke dedicated endpoints that stamp lifecycle timestamps inside task.metadata.lifecycle. UI automatically refreshes via React Query invalidation.

Error handling: Conflicts (HTTP 409) appear as inline alerts if an invalid transition is attempted (e.g., completing a non‑running task).

7. Materials & Equipment (Preview)

These pages surface:

  • Materials picklist (composition from plan + protocol context)
  • Equipment reservation calendar slice (time‑bounded query) Data validation uses Zod schemas client‑side; server responses outside schema are rejected with a descriptive error.

8. Protocols

If a playbook exists, /app/protocols/print renders a print‑optimised view (client fetch: GET /protocol/playbook/{id}). BOM pricing and PDF export remain CLI/API operations for now.

9. System Health

Shows consolidated service + worker status. Use for quick diagnostics before dropping to CLI (docker logs / curl /health).

10. Profile & 2FA

  • Profile: basic account metadata (future: role, capabilities flags)
  • 2FA: Guided setup with QR / secret + verification; toggles enforcement.

11. Error States & Recovery

Common scenarios:

  • 401 / session expiry → automatic redirect to /auth
  • Network timeout (>30s) → abort signal triggers retry suggestion
  • Schema validation failure → inline red banner with raw server fragment (sanitised)

12. Keyboard & UX Conventions

  • Dialogs close with Esc
  • Forms submit with Cmd+Enter (where enabled)
  • Loading states show spinner + verb (“Loading projects…”)—never silent

13. Recommended CLI Complements

Some workflows remain CLI‑only in 0.6.4:

  • Screening & PRISMA exports
  • Protocol costing (bom price)
  • Secrets / policy / provenance export

14. Security Notes

  • No secrets rendered client‑side
  • All writes travel through validated backend (MCP for DB writes)
  • CSRF risk minimised via same‑origin + stateless JWT strategy

15. Roadmap (UI)

Upcoming (post‑0.6.4):

  • Live task & job stream (SSE/WebSocket)
  • Artefact browser (read‑only) with provenance diff
  • Interactive screening triage panel & PRISMA visualisation
  • Protocol PDF export & inline costing view

16. Troubleshooting

Symptom Action
Infinite loading Check /health; verify API base URL in frontend config
Auth loop Clear cookies; restart gateway; verify JWT secret consistency
409 on task start Task already running or completed; refresh page
Empty materials list Ensure approved plan + playbook generated (CLI)

For unresolved issues, consult logs (frontend.log) or raise a ticket.
