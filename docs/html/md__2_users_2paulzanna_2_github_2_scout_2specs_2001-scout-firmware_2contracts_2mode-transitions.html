Mode Transitions & Constraints

Created: 2025-10-25
Branch: 001-scout-firmware
Spec: ../spec.md

Allowed Modes

  • hover (0)
  • loiter (1)
  • forward (2)

Allowed Transitions

  • hover ↔ loiter (bidirectional)
  • loiter → forward (enter forward)
  • forward → loiter (exit forward)
  • forward → hover (emergency exit)

Rejected transitions are responded with STATUSTEXT reason and/or COMMAND_ACK failure.

Constraints & Failure Behavior

  • Global: Mode changes are rejected when arming_allowed=false or required sensors unhealthy.
  • Magnetometer unhealthy: heading-reliant modes (loiter/forward) are unavailable; automatically fall back to hover.
  • Vehicle-launch (moving): forward entry requires enhanced checks; otherwise remain in loiter/hover.
  • CPU budget exceeded: reject transitions that increase load; degrade optional features; no mode entry if cpu_budget_ok=false.

Vehicle-Launch Enhanced Checks (FR-013)

  • Data sources:
    • Speed (km/h) from GPS or odometry; use low-pass filtered estimate.
    • Attitude (roll/pitch) from IMU fusion.
    • Link quality from receiver RSSI/quality metric.
    • Sensor health flags and CPU budget.
  • Windowing & Tolerances:
    • Speed threshold: ≤ 80 km/h over a 1 s moving average.
    • Attitude limits: |roll| ≤ 15°, |pitch| ≤ 15° over a 500 ms window.
    • Link quality: ≥ MIN_LINK_QUALITY (e.g., 50) averaged over 1 s.
    • CPU budget: total ≤ 70% over 1 s window.
  • Outcome:
    • If all pass, arming/mode entry permitted; else rejected with STATUSTEXT.

Precedence with Degrade Policy (Conflict Resolution)

  • Safety and degrade policy take precedence over requested mode.
  • If a required condition becomes invalid mid-flight (e.g., MAG fault while in forward), the system exits forward to loiter/hover per degrade-policy and surfaces a STATUSTEXT reason.
