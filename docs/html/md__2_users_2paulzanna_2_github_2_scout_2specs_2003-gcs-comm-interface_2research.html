COBS Encoding Library Implementation

Decision: Implement custom COBS encoder/decoder optimized for ESP32-S3

Rationale:

  • COBS (Consistent Overhead Byte Stuffing) is a simple algorithm that can be implemented in ~100 lines of C code
  • Custom implementation provides better control over memory allocation and performance optimization
  • Avoids external library dependencies that may not be optimized for embedded constraints
  • Reference implementation available from original COBS specification (Cheshire & Baker, 1999)

Alternatives considered:

  • Third-party COBS libraries: Risk of license conflicts, size overhead, and unnecessary features
  • Alternative framing methods (HDLC, SLIP): More complex, higher overhead than COBS

Implementation approach:

  • Static allocation for encode/decode buffers (no dynamic memory)
  • Streaming decoder to handle partial frames
  • Maximum frame size limit of 512 bytes to bound memory usage

HMAC-SHA256 Crypto Implementation

Decision: Use ESP-IDF's built-in mbedTLS HMAC-SHA256 implementation

Rationale:

  • ESP-IDF includes mbedTLS cryptographic library with hardware acceleration support
  • HMAC-SHA256 is already available and tested in ESP-IDF framework
  • Hardware acceleration on ESP32-S3 provides better performance than software-only implementation
  • Well-tested, industry-standard implementation reduces security risks

Alternatives considered:

  • Custom HMAC implementation: High risk of security vulnerabilities, significant development time
  • Alternative authentication methods (AES-CMAC): Less common, higher complexity
  • Lighter cryptographic primitives: Insufficient security for military threat model

Implementation approach:

  • Use esp_crypto.h APIs for HMAC operations
  • Pre-shared keys stored in NVS with encryption
  • Truncate HMAC-SHA256 to 16 bytes per SCP specification
  • Key rotation support through secure configuration interface

Hardware-in-the-Loop Test Approach

Decision: Staged testing approach combining unit tests, mock GCS simulator, and physical hardware validation

Rationale:

  • Pure protocol logic (framing, crypto, session management) can be unit tested without hardware
  • Mock GCS simulator enables automated integration testing of complete protocol flows
  • Physical hardware testing validates real-world timing and performance constraints
  • Staged approach allows early detection of issues without requiring full hardware setup

Test Strategy:

Unit Tests (Unity framework)

  • Message encoding/decoding correctness
  • COBS framing edge cases
  • HMAC authentication validation
  • Session state transitions
  • Role-based authorization logic

Integration Tests (Mock GCS)

  • Complete protocol handshake sequences
  • Multi-operator conflict resolution
  • Rate limiting enforcement
  • Error response generation
  • Session timeout handling

Hardware Validation

  • Real-time performance measurement (stats command before/after)
  • Multi-transport testing (UART, USB CDC, SPI)
  • Physical security testing (replay attacks, jamming resistance)
  • Multi-operator scenarios with actual GCS clients

Tools:

  • Unity test framework for embedded unit testing
  • Python-based mock GCS simulator for protocol validation
  • Logic analyzer for timing verification
  • Spectrum analyzer for RF interference testing

Flash Storage Limits for Cryptographic Libraries

Decision: mbedTLS HMAC-SHA256 implementation fits within available flash budget

Rationale:

  • ESP32-S3 has 8MB flash, with ~2-3MB available for application code
  • mbedTLS HMAC-SHA256 adds approximately 20-30KB to firmware size
  • Hardware acceleration support reduces code size compared to software-only implementation
  • Flash usage acceptable within current firmware size constraints

Memory Analysis:

  • Current Scout firmware: ~1.2MB
  • Estimated SCP protocol addition: ~50KB (including COBS, session management, CLI commands)
  • mbedTLS HMAC overhead: ~25KB
  • Total increase: ~75KB (well within 2MB application budget)

Alternatives considered:

  • Lighter hash functions (SHA-1, MD5): Insufficient security for military applications
  • Custom crypto implementation: Unacceptable security risk
  • External crypto chip: Adds hardware complexity and cost

Validation approach:

  • Monitor firmware size during development
  • Use ESP-IDF size analysis tools to track memory usage
  • Implement size regression testing in CI pipeline

Protocol Integration Architecture

Decision: Integrate SCP as a new service layer that interfaces with existing safety and audit systems

Rationale:

  • Preserves existing safety architecture and CLI contracts
  • Allows gradual migration from MAVLink without breaking existing functionality
  • Maintains real-time performance by avoiding changes to core control loops
  • Enables A/B testing and fallback to MAVLink if needed

Integration Points:

  • Safety Manager: All GCS commands route through existing safety validation
  • Audit System: Protocol events logged through existing audit infrastructure
  • CLI Framework: New SCP commands follow existing CLI patterns and contracts
  • Telemetry System: SCP telemetry integrates with existing sensor data pipeline

Performance Considerations:

  • Protocol processing runs in separate FreeRTOS task at appropriate priority
  • Message queues prevent blocking of real-time control loops
  • Rate limiting prevents protocol traffic from overwhelming system resources
  • Configurable telemetry rates allow performance tuning for different scenarios
