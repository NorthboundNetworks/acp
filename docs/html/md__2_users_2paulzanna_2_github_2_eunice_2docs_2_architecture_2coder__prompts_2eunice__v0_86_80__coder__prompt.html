AI Coder Prompt — Eunice v0.6.0 “PRISMA‑lite Systematic Review”

Date: 12-08-2025
Role: Senior engineer implementing v0.6.0. Build a reproducible screening pipeline with criteria sets, queues, decisions, and PRISMA flow exports.

0) Mission

Implement a PRISMA‑style, auditable screening pipeline with JSON‑versioned inclusion/exclusion criteria, title/abstract and full‑text screening queues, decision logging, optional active‑learning triage, and PRISMA flow (JSON + SVG) exports.

1) Constraints & Standards

  • Python 3.11+, SQLAlchemy, Alembic, pytest, mypy, ruff.
  • Deterministic counts from decisions; reproducible exports.
  • Minimal local FastAPI optional; CLI/TUI first.
  • All criteria versioned with SHA256; audit every decision (who/when/why).
  • No distribution of publisher PDFs in exports; provenance preserved.

2) Deliverables (DoD‑gated)

  1. Criteria Manager: schema, loader, SHA256 pin, versions table.
  2. Screening Queues: title/abstract and full‑text with batch operations.
  3. Decision Engine: include/exclude/unsure + reason codes, reviewer IDs, timestamps.
  4. Active‑Learning Triage (optional, behind flag): simple relevance model over titles/abstracts; re‑scoring.
  5. PRISMA Flow Builder: node counts and diagram export (JSON + SVG).
  6. Exports: included studies CSV/JSON with canonical IDs and anchor links if available.
  7. Tests & CI enforcing deterministic PRISMA counts and schema validation.

3) Repository Changes

/eunice/screening/ # queues, decisions, criteria manager, AL triage
/eunice/prisma/ # flow builder + exports
/tests/screening/ # unit + snapshot tests
/criteria/schema.json # schema for criteria sets
/criteria/sets/*.json # example criteria

4) Database (Alembic)

Tables:

  • criteria_set(id, name, description)
  • criteria_version(id, set_id, spec_json, sha256, created_at)
  • screening_run(id, run_id, criteria_version_id, stage, metrics_json, created_at)
  • screening_decision(id, screening_run_id, source_id, decision, reason_code, reviewer_id, decided_at)
  • prisma_flow(id, screening_run_id, nodes_counts_json, svg_diagram_path)
  • reviewer(id, name, email)

Indexes: (screening_run_id, source_id) unique; (criteria_version.sha256) unique.

5) Algorithms

5.1 Active‑Learning Triage (optional)

  • Start with keyword‑seeded positives; train TF‑IDF/logistic or embeddings‑cosine baseline.
  • After each batch, re‑score remaining queue; prioritise high‑uncertainty/high‑gain items.
  • Ensure exploration by reserving N% random items per batch.

5.2 PRISMA Counts

  • Deterministic aggregation from screening_decision by stage and reason codes.
  • SVG rendering driven by counts JSON; stable node IDs for snapshot tests.

5.3 Reason Codes

  • Controlled vocabulary: out_of_scope, wrong_population, wrong_outcome, duplication, not_primary_study, language_excluded, etc.

5.4 Inter‑rater (optional)

  • If multi‑reviewer, compute Cohen’s κ on overlap; if κ<τ, flag criteria review.

6) Interfaces

6.1 CLI

  • eunice criteria add --file criteria.json
  • eunice screen ta --run <run_id> --criteria <sha>
  • eunice screen ft --run <run_id> --criteria <sha>
  • eunice decision add --screening <id> --source <sid> --decision include --reason <code>
  • eunice prisma export --screening <id> --out flow.svg
  • eunice export included --screening <id> --format csv

6.2 Internal FastAPI (optional/local)

POST /criteria -> {json} -> {criteria_sha}
POST /screen/ta -> {run_id, criteria_sha} -> {screening_run_id}
POST /screen/ft -> {run_id, criteria_sha} -> {screening_run_id}
POST /decision -> {screening_run_id, source_id, decision, reason_code?, reviewer_id?} -> {ok}
GET /prisma/{id} -> {json, svg}

7) Config & Schemas

  • /criteria/schema.json defines allowed fields and enumerations.
  • Each set at /criteria/sets/*.json; SHA256 stored in DB and manifests.

Example criteria set:

{
"name": "avian-neurons-in-vitro",
"include": {
"population": ["avian", "chicken", "gallus gallus"],
"study_types": ["primary", "protocols"]
},
"exclude": {
"populations": ["in vivo mammalian only"],
"languages": ["non-english"]
},
"notes": "Prefer post-2019 studies; include earlier if foundational."
}

8) Tests (pytest)

  • Schema validation for criteria JSON (positive & negative cases).
  • Deterministic PRISMA: fixed decisions → identical JSON + SVG.
  • AL triage smoke: accuracy improves or uncertainty reduces after two batches.
  • Export integrity: included export equals PRISMA “included” node; IDs/citations valid.

9) CI Gates (fail if below)

  • Criteria schema tests pass; SHA pins recorded.
  • PRISMA JSON/SVG snapshot stable.
  • Export integrity test green.
  • (If AL enabled) triage improves target metric on toy set.

11) Observability & Artefacts

  • Metrics: screened/hour, include rate, reason distribution, κ (if multi‑rater).

12) Risks & Mitigations

  • Inconsistent reviewer decisions → κ monitoring & conflict flags.
  • Criteria drift → versioned criteria with SHA pins; record version in manifests.
  • AL bias → cap model influence; ensure exploration fraction.
