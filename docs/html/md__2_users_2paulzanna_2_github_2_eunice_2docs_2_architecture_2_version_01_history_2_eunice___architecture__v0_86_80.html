Eunice v0.6.0 Architecture — PRISMA‑lite Systematic Review

Date: 14-08-2025
Owner: Paul Zanna
Status: Implemented

1) Executive Summary

v0.6.0 delivers a repeatable screening pipeline following PRISMA‑style methodology: inclusion/exclusion criteria, screening queues, audit logs, PRISMA flow exports (simple + 2020), and included studies exports. An internal active‑learning baseline triage module is present (TF‑IDF + Logistic) and will gain an interactive CLI flag in a minor follow-up.

2) Scope

Goals

  • JSON‑defined criteria sets with versioning and SHA256 pins.
  • Screening queues for title/abstract and full‑text stages.
  • Decision tracking with reasons, timestamps, and reviewer IDs.
  • PRISMA flow diagram export + CSV/JSON data dumps.
  • Optional active‑learning helper that prioritises likely‑relevant papers.

Non‑Goals (now)

  • Complex multi‑user SaaS UI (keep CLI/TUI minimal; local FastAPI optional).
  • Full inter‑rater adjudication UI (basic conflict resolution only).

4) Components

4.1 Criteria Manager

  • Loads criteria JSON; validates against schema; computes SHA256; stores versioned records.

4.2 Screening Queue

  • Maintains per‑stage queues; supports batch operations; records order served and model suggestions.

4.3 Decision Engine

  • Records include|exclude|unsure with reason codes; logs reviewer, timestamp, source hash.
  • Supports conflict handling: if multiple reviewers disagree, mark needs adjudication.

4.4 Active‑Learning Triage (baseline internal)

  • TF‑IDF + Logistic Regression relevance model over titles/abstracts.
  • Re‑scores queue after incremental fitting; prioritises uncertain/high‑gain items (future interactive flag).

4.5 PRISMA Flow Builder

  • Computes counts per PRISMA nodes with two export variants:
    • simple: identified, screened, excluded, full‑text assessed, included (deterministic).
    • 2020: PRISMA 2020 structure including identification, removal before screening (duplicates, automation, other), screening, eligibility, excluded with reason breakdown, and included (deterministic).
  • Data sources:
    • identification and duplicate‑removal are derived from the linked ingestion run via screening_run.run_id → literature_search_runs (result_count, deduped_count). Duplicates = max(result_count - deduped_count, 0).
    • screened/excluded/included and reason breakdowns come from screening_decision for the given screening run.
    • full‑text assessed is set to screened when screening_run.stage = 'full_text' (best‑effort until richer FT tracking lands).
  • Exports JSON + SVG diagram; attaches to run manifest. Outputs are stable for a fixed set of decisions.

4.6 Exports

  • CSV/JSON of included studies with canonical IDs and links to anchors (when available) via helper write_included_export (stable field order).
    • Included export matches PRISMA “included” node exactly for the run (test enforced).

Note: Topic‑level PRISMA summaries may use the existing literature_prisma_view materialized view; per‑run PRISMA uses the screening_run linkage described above.

6) Interfaces

6.1 CLI

  • eunice criteria add --file criteria.json
  • eunice screen ta --run <run_id> --criteria <sha>
  • eunice screen ft --run <run_id> --criteria <sha>
  • eunice prisma export --screening <id> --variant simple|2020 --json --svg --out prisma.svg
  • eunice export included --screening <id> --format csv

6.2 Internal FastAPI (optional/local)

POST /criteria -> {json} -> {criteria_sha}
POST /screen/ta -> {run_id, criteria_sha} -> {screening_run_id}
POST /screen/ft -> {run_id, criteria_sha} -> {screening_run_id}
POST /decision -> {screening_run_id, source_id, decision, reason_code?} -> {ok}
GET /prisma/{id} -> {json, svg}

7) Algorithms & Heuristics

  • Active learning: start with seed positives (keyword heuristics) → train simple relevance model; update after each labelled batch.
  • Reason coding: enumerated controlled vocabulary (e.g., out_of_scope, wrong_population, wrong_outcome, duplication, not_primary_study).
  • Inter‑rater (optional): compute Cohen’s κ on overlap batches; if κ < τ, trigger review of criteria.

9) Observability

  • Metrics: screened/hour, include rate, reason distribution, κ (if multi‑rater), model AUC (if AL used).
  • Artefacts: decisions.csv, prisma.json, prisma.svg, included_export.csv.

10) Security & Licensing

  • Store reviewer identifiers locally only.
  • Preserve provenance; do not redistribute PDF payloads in exports.

11) Performance Targets

  • Title/abstract pass for 1k candidates in ≤ 5 minutes (interactive CLI batching).
  • PRISMA export generated in < 2 seconds from decisions table (+ O(1) lookup of ingestion run for identification/duplicates).

12) Testing & CI

  • Criteria JSON schema validation test.
  • Deterministic PRISMA simple & 2020 JSON counts + SVG structural assertions (counts & reason ordering).
  • Active‑learning improvement test (probability of a known positive increases after second fit).
  • Queue reprioritisation ordering test (scores sorted descending).
  • Included export integrity tests (CSV header, JSON fields parity with decisions count).
  • Helper isolation enables direct unit test of export logic.

13) Migration Plan

  • Add criteria and screening tables; seed example criteria.
  • Wire CLI commands; generate first PRISMA for a golden topic.
  • Backward compatibility: runs without screening remain valid.

14) Risks & Mitigations

  • Inconsistent decisions → κ monitoring and conflict flagging.
  • Criteria creep → versioned criteria with SHA256 pins; record version per run.
  • AL bias → cap model influence; ensure random exploration fraction.

15) Acceptance (DoD)

  • Complete PRISMA flow export (JSON + SVG) from a labelled run (simple and PRISMA 2020 variants).
  • All decisions auditable (who/when/why) and reproducible from DB alone.
  • Included export matches PRISMA “included” node exactly.
