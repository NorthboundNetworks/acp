Code Organization and Module Documentation

Service Architecture Overview

The Scout firmware uses a service-based architecture for the companion mediation, blackbox logging, and gimbal control features. Each service is designed as an independent module with well-defined interfaces and clear separation of concerns.

Core Services (src/services/)

1. Audit Service (audit.c/.h)

Purpose: Companion input mediation and audit trail management
Key Functions:

  • audit_init() - Initialize whitelist and audit buffer
  • audit_process_companion_input() - Main mediation gateway
  • audit_get_recent_entries() - CLI access to audit history
  • audit_set_whitelist() - Configure authorized companion source

Dependencies: SCP GCS gateway (telemetry/audit), safety manager, mode manager
Performance: <500us input evaluation, 8KB memory footprint
Thread Safety: Mutex-protected for multi-threaded access

2. Blackbox Service (blackbox.c/.h)

Purpose: Mission logging with NDJSON format and HMAC signatures
Key Functions:

  • blackbox_init() - FATFS mount and background writer startup
  • blackbox_log_event() - Non-blocking event queuing
  • blackbox_start_mission() - Begin new mission with rotation
  • blackbox_export_mission() - Generate signed export

Dependencies: FATFS, HMAC utilities, sensor aggregation
Performance: <100us queuing latency, 256-event queue capacity
Storage: Keep-last-1 rotation, ~2-5KB per flight minute

3. Gimbal Service (gimbal.c/.h)

Purpose: Two-axis gimbal control with multiple operating modes
Key Functions:

  • gimbal_init() - PWM servo setup and mode initialization
  • gimbal_set_mode() - Switch between fixed/follow/sweep modes
  • gimbal_set_attitude() - Direct angle control
  • gimbal_update() - Background servo updates (50Hz)

Dependencies: LEDC PWM, GPS (for follow mode), timer services
Performance: 50Hz update rate, <300ms command latency
Hardware: GPIO 14 (pitch), GPIO 12 (yaw) servo outputs

CLI Integration (src/cli/)

Each service provides CLI commands following consistent patterns:

  • Status commands show current state and statistics
  • Configuration commands allow parameter adjustment
  • Action commands trigger operations (start/stop/set)
  • Help text provides usage guidance

Command Examples:

audit show --recent 10 # Show last 10 audit entries
blackbox start mission_001 # Begin mission logging
gimbal set -30 45 # Set pitch=-30°, yaw=45°

SCP Gateway (src/services/gcs_gateway_runtime.c)

SCP messages are routed through the gateway to appropriate service handlers:

  • Gimbal control (future): mapped to SCP command set
  • Audit summaries as SCP TLM_AUDIT messages
  • Command acknowledgments via SCP response messages

Utility Libraries (src/lib/)

HMAC Key-Value Store (hmac_kv.c/.h)

  • Secure key storage in ESP32 NVS with hardware encryption
  • Label-based key retrieval with access control
  • Tamper-evident mission log signatures

Feature Configuration (feature_config.h)

  • Centralized performance and safety parameters
  • Queue sizes, rate limits, and timing constants
  • Tuned for ESP32-S3 performance characteristics

Code Quality Standards

Documentation Requirements

  • File headers with purpose, architecture, and performance notes
  • Function documentation with parameters and return values
  • Inline comments for complex algorithms or safety-critical sections
  • Performance characteristics and Constitution compliance notes

Safety Practices

  • Input validation on all external interfaces
  • Buffer overflow protection with size checks
  • Error handling with proper cleanup and logging
  • No blocking operations in real-time code paths

Performance Guidelines

  • Static memory allocation preferred over dynamic
  • Queue-based producer/consumer patterns for background processing
  • Rate limiting to prevent resource exhaustion
  • Performance monitoring and Constitution compliance validation

Thread Safety

  • Mutex protection for shared data structures
  • Lock-free queues for real-time producer paths
  • Clear ownership semantics for mutable state
  • Deadlock prevention through consistent lock ordering

Testing Strategy

Unit Testing

  • Service initialization and configuration
  • Input validation and error handling
  • Performance characterization under load
  • Memory leak detection over extended runs

Integration Testing

  • Cross-service interactions (audit → blackbox → gimbal)
  • CLI command validation with golden transcripts
  • SCP protocol compliance verification
  • End-to-end user story validation

Performance Validation

  • CPU utilization under burst conditions (<70% Constitution limit)
  • Loop timing maintenance (±5% tolerance)
  • Memory usage bounds (no unbounded growth)
  • Flash storage performance and wear characteristics

Maintenance Guidelines

Adding New Features

  1. Follow service-based architecture patterns
  2. Add CLI commands with consistent interface patterns
  3. Include comprehensive documentation and performance notes
  4. Add integration tests and performance validation
  5. Update Constitution compliance documentation

Performance Tuning

  1. Profile with stats command during development
  2. Adjust queue sizes and rate limits in feature_config.h
  3. Validate Constitution requirements after changes
  4. Document performance impact in commit messages

Security Updates

  1. Review input validation for new interfaces
  2. Update threat model and security documentation
  3. Test unauthorized access scenarios
  4. Validate audit logging completeness

This modular architecture ensures maintainable, testable, and performant firmware that meets all Constitution requirements while providing comprehensive mission capabilities.
