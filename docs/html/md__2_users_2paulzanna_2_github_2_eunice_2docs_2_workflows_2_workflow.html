Eunice Research Platform - Application Process Flow

Step-by-Step Process After Research Plan Creation

Phase 1: Research Plan Creation

  1. User Initiates Plan Creation
    • User creates a research topic through the frontend interface
    • User clicks "Generate Research Plan" for the topic
    • Frontend sends POST request to /topics/{topic_id}/ai-plans:enqueue
  2. API Gateway Enqueues Job
    • API Gateway validates the topic exists
    • Enforces single plan per topic constraint (rejects if plan already exists)
    • Creates idempotent job entry in rm_inbox table with type create_research_plan
    • Returns job ID to frontend (202 Accepted status)
  3. Research Manager Processes Job
    • Poller service (poller.py) continuously monitors rm_inbox table
    • Leases available create_research_plan jobs in batches
    • Dispatches job to run_create_plan workflow handler

Phase 2: AI-Powered Plan Generation

  1. Plan Generation Workflow
    • create_plan.py workflow executes with topic data
    • Loads prompt configuration from JSON files (e.g., create_research_plan.json)
    • Calls generate_plan() function in llm.py
  2. LLM Integration
    • Network service receives AI model request via MCP protocol
    • Sends request to external AI provider (OpenAI, Anthropic, etc.)
    • AI generates PlanBrief v1 JSON format with:
      • Sources, outcomes, timeline, key areas
      • Research questions, objectives
      • Structured phases with day estimates
  3. Plan Validation & Storage
    • Generated plan validated against plan_brief_v1.schema.json
    • Plan data normalized (strings converted to arrays, timeline validated)
    • Database mutation creates research plan record via MCP database service
    • Plan stored with canonical format + original PlanBrief v1 in metadata

Phase 3: Task Generation and Assignment

  1. Automatic Task Creation Algorithm Deprecated
    • Database Transaction: All tasks created atomically via MCP database mutation:
      • Plan and tasks created in single transaction using {"tx": True}
      • Each task assigned unique ID and linked to plan via plan_id
      • Initial status set to "pending"
      • Task metadata preserves creation source and basis

  2. Task Worker System Integration

    8.1 Queue Monitoring

    • TaskWorkerService runs continuous monitoring loop (5-second intervals)
    • Uses lease_worker_jobs() to acquire tasks from rm_inbox table
    • Batch processing (up to 3 tasks per cycle) with lease-based concurrency control
    • Heartbeat mechanism maintains task leases during processing

    8.2 Task Classification and Routing

    • Explicit Type Mapping: Direct mapping for specialized task types:
      • process_literature → Literature Worker
      • generate_experiment → Experiment Worker
      • analyze_materials → Materials Worker
      • create_plan → Synthesis Worker
    • Content-Based Classification: For general create_plan tasks, payload analysis determines worker:
      • Keywords ['literature', 'search', 'review'] → Literature Worker
      • Keywords ['experiment', 'design', 'factorial'] → Experiment Worker
      • Keywords ['materials', 'bom', 'cost', 'procurement'] → Materials Worker
      • Default fallback → Synthesis Worker

    8.3 Worker Specializations:

    • Literature Worker:
      • Multi-source academic search (PubMed, Semantic Scholar, arXiv, CORE)
      • Content extraction, filtering, and summarization
      • Citation analysis and reference management
    • Experiment Worker:
      • Design of Experiments (DoE) generation with factorial/screening/response surface templates
      • Statistical power analysis and sample size calculation
      • Protocol creation and experimental timeline planning
    • Materials Worker:
      • Bill of Materials (BOM) generation with cost analysis
      • Multi-supplier sourcing and procurement strategy
      • Risk assessment and inventory management
    • Synthesis Worker:
      • Cross-worker result aggregation and synthesis
      • Multi-format report generation (PDF, HTML, Markdown, JSON, CSV, Excel, PowerPoint)
      • Data visualization and statistical analysis

    8.4 Execution Tracking

    • Each task gets unique execution ID (uuid4()) for tracking
    • Execution records stored in task_executions table with:
      • Worker type and worker ID
      • Start/completion timestamps
      • Progress percentage and current step
      • Error messages and retry count
      • Result data in JSONB format

Phase 4: Task Execution and Processing

  1. Worker Task Processing
    • Tasks progress through statuses: pending → queued → assigned → in_progress → completed
    • Each worker type provides specialized domain expertise:
      • Literature: Multi-source search (PubMed, Semantic Scholar, arXiv)
      • Experiment: DoE generation, power analysis, protocol creation
      • Materials: Cost analysis, supplier management, risk assessment
      • Synthesis: Report generation, visualization, multi-format export
  2. Progress Tracking and Monitoring
    • Task executions tracked in task_executions table
    • Real-time progress updates via WebSocket connections
    • Circuit breaker patterns prevent cascading failures
    • Auto-scaling adjusts worker capacity based on load

Phase 5: Results and Reporting

  1. Result Storage and Aggregation
    • Task results stored in task_results table with metadata
    • Advanced caching system optimizes repeat operations
    • Results available in multiple formats (JSON, PDF, Excel, etc.)
  2. Plan Status Updates
    • Plan completion percentage calculated from task statuses
    • Statistics updated in real-time showing progress metrics
    • Cost tracking includes actual vs. estimated expenses

Phase 6: Research Execution (User-Initiated)

  1. Research Execution Request
    • User can initiate research execution via /topics/{topic_id}/execute-research
    • System validates approved research plan exists
    • Depth configuration applied (basic, detailed, comprehensive)
  2. Context Preparation
    • Research context assembled including topic, plan, and depth settings
    • Task type and parameters configured based on depth level
    • Execution ID generated for tracking
  3. Execution Monitoring
    • Real-time status updates via progress endpoints
    • WebSocket notifications for completion states
    • Error handling with retry mechanisms

Phase 7: Continuous Operations

  1. Health Monitoring
    • All services provide health endpoints (/livez, /readyz, /health)
    • Container health checks ensure service availability
    • Circuit breakers prevent service degradation
  2. Data Persistence
    • Hybrid database pattern: direct reads for performance, MCP writes for consistency
    • PostgreSQL database with hierarchical schema
    • Full CRUD operations with validation and referential integrity
  3. Frontend Integration
    • React frontend with real-time updates
    • shadcn/ui components for consistent UX
    • Progressive loading and error handling

Key Technical Features

  • Idempotent Operations: All critical operations use idempotency keys
  • Fault Tolerance: Circuit breakers, retries, and graceful degradation
  • Scalability: Auto-scaling workers based on demand
  • Performance: Caching, connection pooling, and optimized queries
  • Monitoring: Comprehensive metrics, logging, and health checks
  • Security: JWT authentication, parameterized queries, container hardening

This architecture provides a robust, scalable research platform that automatically progresses from user intent to actionable research plans with intelligent task execution and monitoring.
