Implementation Plan: Custom Firmware for Scout ISR Drone

Branch: 001-scout-firmware | Date: 2025-10-25 | Spec: /specs/001-scout-firmware/spec.md Input: Feature specification from /specs/001-scout-firmware/spec.md

Note: This template is filled in by the /speckit.plan command. See .specify/templates/commands/plan.md for the execution workflow.

Summary

Deliver embedded flight firmware for the Scout ISR platform using PlatformIO and C on ESP32‑S3, with safety‑first arming, deterministic control loops, stable CLI contracts, core sensor integration (MPU‑6050, HMC5883L, BMP388, NEO‑6M), baseline command/telemetry profile, and comprehensive verification across simulation, HITL/SITL, and field tests. Timing and CPU budgets must meet Constitution limits (≤70% total CPU, loop rates within ±5%).

Technical Context

Language/Version: C (GNU C11 where applicable)
Primary Dependencies: ESP‑IDF via PlatformIO; device drivers for MPU‑6050, HMC5883L, BMP388, NEO‑6M; FreeRTOS (bundled)
Storage: ESP‑IDF NVS partition for configuration persistence; default partition 24–64 KB. Namespace cfg.*; total config payload ≤ ~6 KB; single value ≤ 512 B. save performs batched commits with checksum for integrity.
Testing: Multi‑tier: unit (Unity/CMock), integration (CLI golden transcripts), simulation (host mocks/SITL), HITL (ESP32‑S3 dev board), field tests (checklists, logs).
Target Platform: ESP32‑S3 (freenove_esp32_s3_wroom)
Project Type: Single embedded firmware project
Performance Goals: Loop rates within ±5% of configured (p95 loop time captured); total CPU ≤70% in flight; disarm on link‑loss ≤1s (1.0 s detection threshold, disarm ≤1.0 s from detection); telemetry ≥1 Hz for baseline profile with jitter ≤±250 ms and ≥99% delivery over 10 minutes.
Constraints: Safety‑first arming, deterministic real‑time; CLI contract stability; observability via status/stats/blackbox; environmental robustness (MIL‑STD‑810G conditions referenced in README).
Scale/Scope: Single airframe POC with modular sensor stack and baseline command/telemetry profile.

Environmental Constraints Lift-Down (Actionable)

  • Offline-first operation: No network CLI exposure at any time; WiFi/AP disabled in FLIGHT (FR-021). Evidence via integration tests (T048) and CLI transcripts.
  • Power resilience: Safe on boot; starts disarmed; tolerates unexpected power loss without corrupting configuration (dump must remain importable). Evidence: round-trip dump after reboot (T041).
  • Vibration/EMI robustness: Sensor health gating prevents arming on obvious faults; heading modes disabled if MAG unhealthy; altitude hold disabled on BARO anomalies. Evidence: health flags and STATUSTEXT during induced faults (T024, T021, T026).
  • Temperature/environment references: Environmental robustness is a non-functional goal; specific numeric limits are out-of-scope here and will be validated in field tests. Plan requires documenting any observed constraints in quickstart and wiring docs.
  • Deterministic timing under load: Guardrails prevent arming when CPU >70%; features degrade before disarm. Evidence: stats guardrails and bench runs (T042, T043, T044).
