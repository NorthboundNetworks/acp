Sensor Fault Degrade Integration Test

Test ID: T061

Priority: P (Parallel)
User Story: US5 - Sensor Integration, Calibration & Health
Created: 2025-10-25
Type: Integration Test

Purpose

Validate sensor fault handling and degradation policy enforcement during simulated mid-flight sensor failures. This test ensures the system properly detects sensor faults, applies appropriate degradations, and maintains safe operation or performs controlled disarming when required.

Scope

In Scope

  • Sensor fault injection and detection
  • Degrade policy application and mode restrictions
  • Safety gate enforcement during faults
  • STATUSTEXT feedback for sensor faults and degradations
  • Graceful mode transitions and disarming behavior
  • Fault recovery and system restoration

Out of Scope

  • Actual hardware sensor failures (this is simulated testing)
  • Network latency effects on fault detection
  • Long-term wear and drift characteristics
  • Environmental stress testing

Test Environment

Target System

  • ESP32-S3 Scout firmware with sensor fault simulation capability
  • CLI interface for injecting simulated sensor faults
  • Serial connection for monitoring system behavior

Test Harness

  • Python script for automated fault injection
  • CLI command sequences for manual testing
  • Telemetry monitoring for system responses

Prerequisites

  • T059: Degrade policy implementation
  • T060: Safety manager integration with degrade policy
  • Working CLI interface for sensor fault simulation
  • Functional telemetry and STATUSTEXT reporting

Test Cases

TC001: IMU Fault Response

Objective: Verify IMU fault triggers immediate disarm when armed

Setup:

  1. System armed and in HOVER mode
  2. All sensors initially healthy
  3. Telemetry monitoring active

Procedure:

  1. Inject IMU fault via CLI: sensor_fault imu fault
  2. Monitor system response within 1 second
  3. Verify immediate disarm occurs
  4. Check STATUSTEXT for fault notification
  5. Attempt re-arming (should be blocked)
  6. Clear fault: sensor_fault imu ok
  7. Verify arming becomes available

Expected Results:

  • System disarms within 1 second of IMU fault
  • STATUSTEXT: "IMU fault - emergency disarm"
  • Arming blocked while fault persists
  • Recovery successful after fault clearance

Pass Criteria: IMU fault triggers immediate disarm and recovery works

TC002: Magnetometer Fault Degradation

Objective: Verify MAG fault restricts heading modes but allows attitude modes

Setup:

  1. System armed and in LOITER mode (heading-reliant)
  2. All sensors initially healthy

Procedure:

  1. Inject MAG fault via CLI: sensor_fault mag fault
  2. Verify system exits LOITER to HOVER mode
  3. Check STATUSTEXT for mode restriction notification
  4. Attempt to enter FORWARD mode (should be rejected)
  5. Verify HOVER mode remains available
  6. Clear fault: sensor_fault mag ok
  7. Verify heading modes become available again

Expected Results:

  • Automatic transition from LOITER to HOVER
  • STATUSTEXT: "Sensor faults: heading modes disabled"
  • FORWARD/LOITER mode requests rejected
  • HOVER mode remains functional
  • Full functionality restored after fault clearance

Pass Criteria: MAG fault properly degrades to attitude-only modes

TC003: Barometer Fault Handling

Objective: Verify BARO fault disables altitude hold but maintains other modes

Setup:

  1. System armed and in LOITER mode
  2. All sensors initially healthy

Procedure:

  1. Inject BARO fault via CLI: sensor_fault baro fault
  2. Monitor for altitude hold disable notification
  3. Verify flight modes remain available (no automatic transition)
  4. Check that altitude-dependent features are disabled
  5. Clear fault and verify restoration

Expected Results:

  • STATUSTEXT: "Sensor faults: altitude hold disabled"
  • Flight modes remain available
  • Altitude hold functionality disabled
  • Recovery successful after fault clearance

Pass Criteria: BARO fault disables altitude features without forcing mode changes

  1. Inject GPS fault via CLI: sensor_fault gps fault

  • Current flight mode maintained

Pass Criteria: GPS fault gracefully degrades without mode changes

TC005: CPU Budget Exceeded Response

Objective: Verify CPU overload triggers feature reduction and eventual arming block

Setup:

  1. System initially within normal CPU budget
  2. Monitoring CPU utilization via telemetry

Procedure:

  1. Trigger CPU overload via CLI: cpu_overload simulate 75
  2. Monitor for feature reduction (telemetry rate decrease)
  3. Increase load further: cpu_overload simulate 85
  4. If armed, verify critical overload triggers disarm
  5. If disarmed, verify arming is blocked
  6. Clear overload: cpu_overload clear
  7. Verify normal operation restoration

Expected Results:

  • 75% CPU: STATUSTEXT "CPU degrade: reducing telemetry rate"
  • 85% CPU: Disarm if armed, arming blocked if disarmed
  • Feature reduction active during overload
  • Recovery after load reduction

Pass Criteria: CPU overload triggers appropriate degradations and safety responses

TC006: Multiple Simultaneous Faults

Objective: Verify system handles multiple sensor faults appropriately

Setup:

  1. System armed and in LOITER mode
  2. All sensors initially healthy

Procedure:

  1. Inject MAG fault: sensor_fault mag fault
  2. Inject BARO fault: sensor_fault baro fault
  3. Verify system degrades to HOVER with multiple restrictions
  4. Inject IMU fault: sensor_fault imu fault
  5. Verify immediate disarm (most severe fault wins)
  6. Clear all faults systematically
  7. Verify full recovery

Expected Results:

  • MAG+BARO: Multiple degradations, transition to HOVER
  • IMU added: Immediate disarm (critical fault)
  • STATUSTEXT indicates multiple fault types
  • Systematic recovery possible

Pass Criteria: Multiple faults handled with appropriate severity prioritization

Fault Injection Test (test_sensor_fault_degrade.py)

#!/usr/bin/env python3
"""
Sensor Fault Degradation Integration Test
Simulates sensor failures and validates system responses
"""
import time
import json
import serial
import subprocess
from datetime import datetime
class SensorFaultTester:
def __init__(self, serial_port='/dev/ttyUSB0', baudrate=115200):
self.serial_port = serial_port
self.baudrate = baudrate
self.connection = None
def connect(self):
"""Connect to Scout firmware CLI"""
try:
self.connection = serial.Serial(self.serial_port, self.baudrate, timeout=1)
time.sleep(2) # Allow connection to stabilize
return True
except Exception as e:
print(f"Connection failed: {e}")
return False
def send_cli_command(self, command):
"""Send CLI command and capture response"""
if not self.connection:
return None
self.connection.write(f"{command}\n".encode())
time.sleep(0.5) # Allow command processing
response = ""
while self.connection.in_waiting:
response += self.connection.read(self.connection.in_waiting).decode()
time.sleep(0.1)
return response.strip()
def inject_sensor_fault(self, sensor, fault_type):
"""Inject sensor fault via CLI"""
command = f"sensor_fault {sensor} {fault_type}"
return self.send_cli_command(command)
def get_system_status(self):
"""Get current system status"""
return self.send_cli_command("status")
def test_imu_fault_disarm(self):
"""TC001: IMU Fault Response"""
print("Testing TC001: IMU Fault Response")
# Ensure system is armed first
self.send_cli_command("arm")
time.sleep(1)
# Check initial status
initial_status = self.get_system_status()
# Inject IMU fault
fault_response = self.inject_sensor_fault("imu", "fault")
# Monitor for disarm within 1 second
time.sleep(1.5)
post_fault_status = self.get_system_status()
# Try to arm (should be blocked)
arm_attempt = self.send_cli_command("arm")
# Clear fault
clear_response = self.inject_sensor_fault("imu", "ok")
time.sleep(1)
# Try to arm again (should work)
recovery_arm = self.send_cli_command("arm")
return {
'test': 'imu_fault_disarm',
'initial_status': initial_status,
'fault_response': fault_response,
'post_fault_status': post_fault_status,
'arm_blocked': 'blocked' in arm_attempt.lower() or 'reject' in arm_attempt.lower(),
'recovery_successful': 'success' in recovery_arm.lower() or 'arm' in recovery_arm.lower()
}
def test_mag_fault_degrade(self):
"""TC002: Magnetometer Fault Degradation"""
print("Testing TC002: MAG Fault Degradation")
# Set to LOITER mode
self.send_cli_command("arm")
self.send_cli_command("mode loiter")
time.sleep(1)
initial_mode = self.send_cli_command("status")
# Inject MAG fault
fault_response = self.inject_sensor_fault("mag", "fault")
time.sleep(2)
# Check if mode changed to HOVER
post_fault_status = self.get_system_status()
# Try to enter FORWARD mode (should be rejected)
forward_attempt = self.send_cli_command("mode forward")
# Clear fault and test recovery
clear_response = self.inject_sensor_fault("mag", "ok")
time.sleep(1)
recovery_status = self.get_system_status()
return {
'test': 'mag_fault_degrade',
'initial_mode': initial_mode,
'fault_response': fault_response,
'post_fault_status': post_fault_status,
'forward_blocked': 'reject' in forward_attempt.lower(),
'recovery_status': recovery_status
}
def run_all_tests(self):
"""Execute complete test suite"""
if not self.connect():
return {'error': 'Failed to connect to firmware'}
try:
results = {
'tc001_imu_fault': self.test_imu_fault_disarm(),
'tc002_mag_fault': self.test_mag_fault_degrade(),
# Additional tests would be added here
'timestamp': datetime.now().isoformat()
}
return results
finally:
if self.connection:
self.connection.close()
def main():
tester = SensorFaultTester()
results = tester.run_all_tests()
print(json.dumps(results, indent=2))
# Simple pass/fail analysis
passed = 0
total = 0
for test_name, test_result in results.items():
if isinstance(test_result, dict) and 'test' in test_result:
total += 1
# Simple heuristics for pass/fail
if test_result.get('recovery_successful') or test_result.get('forward_blocked'):
passed += 1
print(f"\nSensor Fault Tests: {passed}/{total} passed")
return 0 if passed == total else 1
if __name__ == "__main__":
import sys
sys.exit(main())
